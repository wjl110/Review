## JD 关键词知识点详解

针对岗位 JD 涉及的专业术语和可能需要补充的知识，以下是详细解释：

### 1. 创意工具品类（Creative Tools）

**3D 打印（3D Printing）**
- **原理**：FDM（熔融沉积成型，家用常见，如拓竹）、SLA/DLP（光固化，精度高）、SLS（选择性激光烧结，工业级）
- **工作流程**：建模（CAD/Blender）→ 切片（Cura/PrusaSlicer）→ 打印 → 后处理
- **用户痛点**：调平难、支撑难拆、打印失败率高、材料选择复杂
- **代表产品**：拓竹（Bambu Lab）、创想三维（Creality）、Prusa

**UV 打印（UV Printing）**
- **原理**：紫外线固化墨水，可在多种材质表面直接打印
- **应用场景**：手机壳定制、亚克力标牌、木板/金属印刷
- **与3D打印区别**：UV是表面印刷（2.5D），3D是立体成型

**激光切割/雕刻（Laser Cutting/Engraving）**
- **类型**：CO2 激光（切割亚克力/木板）、光纤激光（金属）、二极管激光（入门级）
- **工作流程**：矢量图设计（SVG/DXF）→ 参数设置（功率/速度）→ 切割/雕刻
- **代表产品**：xTool、Glowforge、K40

**CNC 雕刻（CNC Routing）**
- **原理**：计算机数控，铣刀切削材料
- **适用材料**：木材、亚克力、软金属、PCB 电路板
- **优势**：精度高、可加工金属，缺点噪音大、速度慢

### 2. AI 创作范式（AI Creative Paradigm）

**Vibe Design / Vibe Coding**
- **概念**：2024-2025 兴起的 AI 驱动设计/编程范式，用户用自然语言描述意图，AI 生成设计稿/代码
- **代表工具**：
  - **Lovart**：AI 设计工具，文本生成视觉设计（海报、UI、品牌物料）
  - **v0.dev**：Vercel 的 AI 生成 UI 工具
  - **Cursor/Windsurf**：AI 编程 IDE
- **与传统设计工作流对比**：
  - 传统：需求 → 草图 → 设计稿 → 评审 → 修改（周期天级）
  - Vibe Design：描述需求 → AI 生成多版本 → 选择/微调 → 导出（分钟级）

**AI 绘图工具对比**

| 工具 | 厂商 | 特点 | 适用场景 |
|-----|------|------|---------|
| MidJourney | 独立 | 艺术感强、美学风格突出 | 概念艺术、插画 |
| Stable Diffusion | 开源 | 可控性强、可本地部署、LoRA 微调 | 定制化生产、商业应用 |
| DALL-E 3 | OpenAI | 语义理解强、遵循提示词准确 | 营销物料、图文匹配 |
| 即梦 (Jimeng) | 字节 | 中文支持好、视频生成强 | 短视频、国内用户 |
| 可灵 (Kling) | 快手 | 视频生成强、物理模拟好 | 视频创作 |
| Flux | Black Forest Labs | 开源 SOTA、细节表现好 | 高质量图像生成 |

**ComfyUI 工作流**
- **概念**：基于节点的可视化 AI 生成工作流编排工具
- **优势**：
  - 流程透明：每个步骤可见可控
  - 可复用：工作流可保存分享
  - 灵活：可组合多种模型、LoRA、ControlNet
- **典型工作流**：文生图 → 图生图 → 放大修复 → 局部重绘

**LoRA / ControlNet**
- **LoRA**：Low-Rank Adaptation，低秩适应，用于模型微调，保持基础模型能力的同时学习特定风格/角色
- **ControlNet**：控制网络，通过边缘/姿态/深度图等控制 AI 生成结构，解决 AI 生成「不听话」问题

### 3. 多模态内容生成管线

**文生图（Text-to-Image）→ 图生视频（Image-to-Video）→ 视频剪辑**
- **文生图**：Prompt Engineering 决定输出质量，关键技巧包括权重语法、负面提示词、采样器选择
- **图生视频**：首尾帧控制、运动幅度控制、物理一致性
- **角色一致性**：IPAdapter、InstantID 等技术保持人物/物体在多镜头中的一致性

**播客生成（AI Podcast/Audio）**
- **语音克隆**：基于 10-30 秒音频样本克隆音色
- **对话生成**：大模型生成双人/多人对话脚本
- **音频合成**：TTS 引擎合成带情感、语调的音频

### 4. 智能硬件与软件协同

**桌面端软件（Desktop Application）**
- **切片软件**：Cura、PrusaSlicer、Bambu Studio，将 3D 模型转为打印机可识别的 G-code
- **设计软件**：Blender（开源 3D）、Fusion 360（CAD）、LightBurn（激光）
- **平台特性**：本地算力利用、硬件驱动对接、离线可用性

**Web 端与桌面端协同**
- **云端切片**：复杂模型上传云端切片，降低本地性能要求
- **模型库**：云端模型管理，一键推送打印
- **社区/市场**：设计分享、付费模型交易

# 三、模拟问答库（一面高频 5 题）

## 问题 1：项目深挖 - RAG 智能体项目

### 面试官提问
"你简历里提到做了微博内部的 RAG 智能体，帮我详细讲讲这个项目的完整流程。你们是如何做知识切块的？Embedding 用的什么模型？幻觉控制怎么做的？"

### 面试官意图
- 考察对 RAG 全链路的技术理解深度
- 验证是否真动手做过，还是只是参与
- 看数据思维和迭代能力

### 建议答法要点（bullet）

**项目背景与目标**
- 微博内部知识分散在 Wiki、Confluence、各种文档，员工检索效率低
- 目标：让员工用自然语言提问，快速获取准确答案
- 我的角色：产品负责人，从 0 到 1 主导需求梳理、原型设计、上线迭代

**知识切块策略**
- 按语义边界切块，而非固定长度：段落级 → 句子级 → 代码块/表格特殊处理
- 重叠窗口：前后 20% 内容重叠，防止上下文割裂
- 元数据注入：给每块打上文档来源、更新时间、作者标签，便于溯源

**Embedding 与向量库**
- 模型选择：测试过 BGE-large、M3E、OpenAI Ada-002，最终选用 **BGE-large-zh**（中文场景性价比最优）
- 向量数据库：Milvus，支持动态扩缩容，亿级向量检索延迟 < 100ms
- 索引策略：HNSW + IVF 混合，平衡召回率和速度

**幻觉控制**
- 检索增强：Top-K 检索结果必须相关性得分 > 0.75 才进入 LLM 上下文
- 引用溯源：强制要求 LLM 在回答中标注信息来源文档
- 置信度阈值：不确定时触发 "我需要更多信息" 而非瞎编
- 人工反馈闭环：收集用户 thumbs up/down，Bad Case 每周复盘

**数据与迭代**
- 周期内 1500+ 人次使用，收集 58 条反馈
- A/B 测试：对比切片策略（语义 vs 固定长度），准确率提升约 12%
- 上线 GitHub 社区，获得一定 star 和 issue 反馈

### 追问深度问题

**追问 1**: "如果要把这套 RAG 系统迁移到 eufyMake 的创作知识库（比如 3D 打印故障排查、材料参数），你会做哪些调整？"

- **答法要点**:
  - 知识形态差异：从文档型 → 结构化数据（材料参数表、故障码对应表）
  - 引入 SQL/图谱混合检索：精确查询打印温度、速度等数值型参数
  - 多模态扩展：支持用户上传打印失败照片，视觉检索匹配相似故障案例
  - 用户分层：新手用对话式问答，进阶用户用 DSL 精准查询

**追问 2**: "RAG 和 Fine-tuning 怎么选？eufyMake 如果要做一个 AI 设计助手，你会怎么选？"

- **答法要点**:
  - RAG 适合：知识更新频繁、需要溯源、多领域知识
  - Fine-tuning 适合：需要特定风格/格式输出、任务边界清晰
  - eufyMake 场景：混合方案 —— RAG 提供知识底座（材料库、故障库），Fine-tuning 优化设计建议的语气/风格

### 反面回答（避免什么）
- **避免**：只讲功能不讲技术选型逻辑（"我们用了向量数据库"→ 为什么选它？）
- **避免**：不提失败或迭代（RAG 上线必有幻觉问题，要诚实讲怎么解决）
- **避免**：混淆参与和主导（"我们团队做了" → 面试官会问 "你具体做了什么"）

## 问题 2：AI 创作工具深度

### 面试官提问
"Boss 上你提到用过 Lovart、即梦、可灵这些工具。以 Lovart 为例，你觉得它相比传统设计工具（Figma/PS）的核心差异是什么？如果让你设计 eufyMake 的 AI 设计功能，你会借鉴 Lovart 的哪些点？"

### 面试官意图
- 考察是否真用过这些工具，还是只是 "了解"
- 看产品抽象能力：从工具使用 → 设计思路提炼
- 验证对 AI 创作范式的理解

### 建议答法要点（bullet）

**Lovart 核心差异分析**
- **输入方式**：自然语言描述 → 多版本设计稿，而非从空白画布开始
- **迭代模式**：选择 → 微调 → 再生成，而非手动绘制每一个元素
- **成本结构**：时间成本从小时级降到分钟级，试错成本大幅降低
- **用户门槛**：不需要掌握设计软件，但需要掌握 "描述能力"（Prompt Literacy）

**Lovart 可借鉴之处**
- **多版本并行生成**：一次 Prompt 出 3-4 个方案，降低用户选择焦虑
- **视觉反馈闭环**：生成后支持区域重绘、风格迁移、尺寸适配
- **模板工作流**：从描述 → 品牌规范 → 多场景物料（海报/社媒/视频封面）

**eufyMake 设计功能设想**
- **Vibe Design 接入**：用户描述 "我想要一个赛博朋克风格的手机支架" → AI 生成 3D 模型预览 + 打印参数建议
- **风格迁移**：上传参考图片 → 提取风格 → 应用到用户自己的模型
- **可制造性检查（DFM）**：AI 生成模型后自动检测悬垂角度、壁厚是否适合打印

### 追问深度问题

**追问 1**: "3D 打印场景下，AI 生成模型面临什么独特挑战？如何解决？"

- **答法要点**:
  - **水密性（Watertight）**：AI 生成的 mesh 常有破面，需要后处理修复
  - **支撑结构**：复杂结构需要可打印性检查，AI 应给出支撑建议
  - **耗材匹配**：不同材料（PLA/ABS/PETG）对模型结构有不同要求
  - **解决思路**：生成后接切片软件 API 做预检，不通过则反馈给用户调整

**追问 2**: "你用过 ComfyUI，如果要在 eufyMake 里落地一个类似的工作流编辑器给创作者，你会怎么设计？"

- **答法要点**:
  - 用户分层：新手用预设模板，进阶用户用节点编辑
  - 节点设计：抽象为「输入 → 处理 → 输出」三层，隐藏底层参数
  - 错误反馈：节点连线错误时给出可视化提示，而非报错堆栈
  - 社区共享：工作流可分享，类似 ComfyUI 的 JSON 导出机制

### 反面回答（避免什么）
- **避免**：只谈功能不谈 trade-off（"加这个功能很好" → 开发成本呢？维护成本呢？）
- **避免**：对工具理解停留在表层（" Lovart 能 AI 做设计 " → 具体什么场景？生成质量如何？）
- **避免**：脱离用户场景谈技术（先做用户调研，再决定 AI 功能边界）

|-----|------|------|---------|

- **定量**：问卷投放（现有用户 + 潜在用户），覆盖使用频次、痛点排序、付费意愿

### 追问深度问题

**追问 1**: "3D 打印的核心痛点是打印失败率高，从产品经理角度，你会怎么降低这个失败率？"

## 问题 4：失败与迭代

### 面试官提问
"你简历里提到收集 58 条反馈驱动迭代。能讲一个具体的失败案例吗？当时怎么发现的，怎么解决的，最后结果如何？"

### 面试官意图
- 考察抗压能力和 problem-solving
- 看数据驱动决策的真实案例
- 验证是否诚实（每个人都有失败，敢说才是真自信）

### 建议答法要点（bullet）

**案例选择：RAG 智能体早期幻觉问题严重**

**失败现象**
- 上线第一周，用户反馈 "AI 胡编乱造" 占比 35%
- 典型案例：问 "微博视频号收益政策"，AI 给出 2023 年的旧政策，未提及 2024 年新政

**发现过程**
- 埋点数据：thumbs down 率 40%+，远高于预期 10%
- 用户访谈：运营同事抱怨 "不敢直接引用 AI 回答，还要人工核对"
- 根因分析：知识库更新不及时，RAG 检索到旧文档，LLM 未做时效判断

**解决方案**
- **短期（1 周）**：增加 "知识截止日期" 提示，对政策类问题加 "请核实最新版本" 免责声明
- **中期（2 周）**：接入文档更新时间戳，优先返回最近 3 个月文档
- **长期（1 月）**：建立知识库自动同步机制，源文档更新后 24h 内同步到向量库

**结果**
- thumbs down 率从 40% → 12%
- 运营团队使用频次提升 2 倍
- 沉淀了 "时效敏感型知识" 的处理 SOP

**我的反思**
- 一开始过度关注技术实现，忽视了内容运营
- RAG 产品 = 技术 + 内容 + 运营，缺一不可
- 用户反馈要分层：哪些问题能靠产品解决，哪些需要运营规范

### 追问深度问题

**追问 1**: "如果今天重来这个 RAG 项目，你会在哪个环节做得不一样？"

- **答法要点**:
  - 前期：做更充分的知识库 audit，上线前清理过期文档
  - 中期：建立 "回答置信度" 可视化，让用户知道 AI 是否确定
  - 后期：引入专家审核机制，关键业务知识人工抽检

**追问 2**: "eufyMake 如果要做 AI 设计功能，你觉得最大的风险点可能是什么？"

- **答法要点**:
  - **生成质量不稳定**：用户用同样的描述，两次结果差异大 → 影响信任
  - **版权风险**：AI 生成模型与现有设计相似，引发法律问题
  - **打印失败**：AI 生成模型好看但打不出来 → 用户挫败感强
  - **应对**：质量上引入种子锁定 + 版本控制；版权上训练数据清洗 + 相似度检测；打印上做可制造性检查

### 反面回答（避免什么）
- **避免**：把失败包装成成功（"虽然失败了但学到很多" → 太套路）
- **避免**：推卸责任（"主要是技术团队..." → 产品owner 要背锅）
- **避免**：不说数据（"用户反馈不好" → 具体多少？怎么度量？）

## 问题 5：协作冲突与跨团队沟通

### 面试官提问
"做 iOS 系统级合作时，你提到要协调研发、算法、UX、运营多个团队。遇到过最大的冲突是什么？怎么解决的？"

### 面试官意图
- 考察跨职能协作能力（eufyMake 也需要硬软协同）
- 看沟通技巧和 owner 意识
- 验证是否理解技术约束和商业目标的平衡

### 建议答法要点（bullet）

**冲突案例：Spotlight 索引频率的技术限制 vs 用户体验**

**背景**
- 产品需求：Spotlight 搜索微博内容要 "准实时"，用户发博后 5 分钟内可搜到
- 技术反馈：iOS Spotlight API 有索引频率限制，频繁更新会触发系统限流

**冲突点**
- 产品/运营：实时性是核心卖点，延迟影响体验
- 研发团队：技术限制客观存在，强行突破可能导致 App 被系统降权
- 我的位置：中间协调，需要找到双方都能接受的方案

**解决过程**
- **第一步：数据说话**
  - 分析用户行为数据：80% 的搜索发生在发博后 1 小时内
  - 区分内容类型：热搜/大 V 内容需要实时，普通用户内容可容忍延迟
- **第二步：分层方案**
  - 高优先级内容（热搜、认证用户）：5 分钟延迟，专用索引通道
  - 普通内容：30 分钟延迟，批量索引
  - 技术方案：客户端本地缓存 + 云端增量更新结合
- **第三步：A/B 验证**
  - 灰度 5% 用户测试分层方案，对比全实时（模拟）的搜索成功率
  - 结果：分层方案搜索成功率 98%，全实时方案因限流实际成功率只有 85%
- **第四步：沟通落地**
  - 和技术团队：确认方案可行，排期 2 周
  - 和运营团队：解释技术限制，用数据说明分层方案用户体验更好
  - 向上同步：给总监汇报，获得资源支持

**结果**
- 方案顺利上线，日均使用量 3 万+
- 建立了 "高优内容分级" 的协作机制，后续项目复用

### 追问深度问题

**追问 1**: "如果 eufyMake 的研发团队说 'AI 生成 3D 模型技术上还做不到商用质量'，你会怎么推进？"

- **答法要点**:
  - 先理解技术边界：是生成质量不行，还是后处理修复成本高？
  - 降级方案：AI 生成草图 → 人工精修 → 打印，而非一步到位
  - 分阶段落地：
    - Phase 1：AI 辅助设计（参数建议、自动修复）
    - Phase 2：简单几何体生成（盒子、支架）
    - Phase 3：复杂模型生成（待技术成熟）
  - 证据支撑：拿竞品案例（如 Blacksmith.ai、Meshy.ai）说明技术可行性

**追问 2**: "你和算法团队的沟通方式，和与客户端团队有什么不同？"

- **答法要点**:
  - **算法团队**：需要明确的技术指标（准确率、延迟、召回率），用论文/业界方案做 reference，关注数据 pipeline
  - **客户端团队**：更关注交互细节、性能、兼容性，用原型/交互稿沟通更高效
  - **共同**：都需要背景上下文（为什么做、用户是谁、成功标准）

### 反面回答（避免什么）
- **避免**：只说 "沟通协调" 不提具体方法（"我约大家开会" → 怎么开的？达成了什么？）
- **避免**：否定技术团队（"他们不懂业务" → 显得不专业）
- **避免**：不提数据或证据（"我觉得用户会喜欢" → 为什么觉得？）

## 一面常见 5 条减分点与纠正

| 减分点 | 反面案例 | 纠正方式 |
|-------|---------|---------|
| **1. 回答过长无重点** | "这个项目是这样...然后那样...后来又..." → 3 分钟没讲完背景 | **纠正**：用 STAR 框架，每个部分控制在 3 句话内，主动问 "需要我在哪个环节展开？" |
| **2. 技术术语堆砌** | "我们用了 RAG、Embedding、HNSW、向量库..." → 没讲清为什么选 | **纠正**：技术选型要讲 trade-off，"我们对比了 X 和 Y，因为 Z 原因选了 X" |
| **3. 只说成功不说失败** | "项目很顺利，上线效果很好" → 显得不真实 | **纠正**：主动讲一个失败及解决，"最大的坑是 X，我们发现是因为 Y，最后通过 Z 解决" |
| **4. 对岗位理解模糊** | "我觉得这个岗位是做大模型的吧" → JD 都没读透 | **纠正**：面试前把 JD 读 3 遍，用一句话概括岗位核心，开场就说出对 eufyMake 的理解 |
| **5. 反问环节问薪资/加班** | "请问加班多吗？薪资能给多少？" → 一面不合适 | **纠正**：一面反问聚焦业务、团队、技术栈，薪资/加班留到 HR 面或二面 |

## Vibe Design 工具详解

### Lovart 核心功能

**定位**：AI 驱动的视觉设计工具，主打 "自然语言生成设计"

**主要功能模块**：
1. **文生设计**：输入描述生成海报、社媒图、品牌物料
2. **多版本并行**：一次生成 3-4 个方案供选择
3. **智能编辑**：区域重绘、风格迁移、智能扩图
4. **模板工作流**：从描述 → 多场景物料批量生成

**与同类工具对比**：

| 工具 | 优势 | 劣势 |
|-----|------|------|
| Lovart | 视觉质量高、操作简单、适合社媒物料 | 精细化编辑能力弱于传统工具 |
| MidJourney | 艺术感强、风格多样 | 可控性差、需要 Discord |
| Canva AI | 与模板库结合好、商业素材多 | 生成质量一般、创意受限 |
| Figma AI | 与设计 workflow 结合 | 需有设计基础 |

### 面试可用观点

> "我用 Lovart 做个人产品的视觉设计，感受最深的是它改变了设计的工作流 —— 从 '先学软件再创作' 变成 '先描述想法再微调'。但局限性也很明显：对精准的品牌规范（如 LOGO 位置、字体层级）控制还不够。eufyMake 如果要引入 Vibe Design，我觉得可以在 '创意发散' 环节用 AI，在 '精确控制' 环节保留传统工具的优势，做混合工作流。"
