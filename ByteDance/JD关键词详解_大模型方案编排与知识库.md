## 二、「大模型方案编排及设计」详解

### 什么是「编排」（Orchestration）

编排 = 设计大模型完成任务的**工作流程（Workflow）**，不是让模型"裸跑"，而是：

```
输入（商品图文/直播片段）
    ↓
[节点1: 意图识别] → 这是商品描述？直播口播？用户评论？
    ↓
[节点2: 路由分发] → 不同内容类型进不同处理链路
    ↓
[节点3: 知识检索] → 从规则库/知识库召回相关条款
    ↓
[节点4: 模型判断] → 大模型基于检索结果做初步判断
    ↓
[节点5: 规则过滤] → 硬规则兜底（关键词黑名单等）
    ↓
[节点6: 置信度分层] → 高置信自动过/中置信人工审/低置信拦截
    ↓
[节点7: 结果输出] → 通过/拒绝/需补充材料
```

### 电商场景的典型编排示例

**场景：直播口播实时审核**

```
[输入] 主播口播语音（实时转文字）
    ↓
[节点1: ASR转文字] → "这款面膜用了立即美白，效果绝对第一"
    ↓
[节点2: 意图识别] → 商品功效宣传
    ↓
[节点3: 规则库检索] → 召回《化妆品功效宣传规范》
    ↓
[节点4: 关键词匹配] → 命中"立即""绝对""第一"等极限词
    ↓
[节点5: 大模型判断] → "该描述涉嫌夸大功效，违反广告法第X条"
    ↓
[节点6: 风险分级] → 高风险（功效夸大）
    ↓
[节点7: 处置决策] → 实时弹窗提醒主播 → 同步人工审核队列 → 记录信用分
```

### 「编排」vs「写Prompt」的区别

| 维度 | 写Prompt | 编排Workflow |
|------|----------|--------------|
| **粒度** | 单次对话的指令 | 完整业务流程的节点设计 |
| **复杂度** | 一段文本 | 多节点+条件分支+循环 |
| **交付物** | Prompt模板 | Workflow DAG图 + 节点配置 |
| **工具** | 文本编辑器 | 扣子Coze/Dify/LangChain等 |
| **你的角色** | 写指令的人 | 设计流程的产品/运营 |

### 候选人关联经验

**你有的**：AIFlow自动化发博Workflow
```
[输入]博文主题
    ↓
[节点1]意图识别（热点/日常/推广）
    ↓
[节点2]内容检索（搜索热门内容）
    ↓
[节点3]多模型生成
    ↓
[节点4]质量评估
    ↓
[节点5]人工审核（钉钉推送）
    ↓
[节点6]自动发布
```

**迁移点**：内容生成Workflow → 内容审核Workflow
- 相同：都有意图识别、都有质量评估、都有人工审核节点
- 差异：审核Workflow容错率更低（需多层校验）、时效性要求更高（秒级响应）

## 三、「规则库/知识库/案例库」详解

这是RAG（检索增强生成）在电商审核场景的**三种知识类型**，分别解决不同问题：

### 1. 规则库（Rule Base）

**定义**：明确的判定规则，通常是**结构化的、无歧义的、必须严格执行**的。

**电商审核场景示例**：
```
规则库条目示例：
├── 极限词规则
│   ├── "第一""最佳""顶级" → 高风险
│   ├── "绝对""永久""100%" → 中风险
│   └── 行业限定词（如"国家级"需资质证明）
├── 禁售品规则
│   ├── 药品类（需许可证）
│   ├── 医疗器械（需备案）
│   └── 违禁品（毒品、枪支等，直接封禁）
├── 图片合规规则
│   ├── 二维码/联系方式（需审核）
│   ├── 侵权素材（品牌LOGO盗用）
│   └── 涉黄涉暴（直接拦截）
└── 价格规则
    ├── 划线价需有真实销售记录
    └── 促销价不得低于成本价（防止倾销）
```

**技术形态**：
- 关键词黑名单（正则匹配）
- 结构化规则引擎（Drools/自研）
- 决策树（if-then-else）

**使用方式**：硬规则兜底，命中即触发相应动作（拦截/告警/人工复核）。

### 2. 知识库（Knowledge Base）

**定义**：供模型**理解背景、参考依据**的文档集合，通常是**非结构化的、需要语义检索的**。

**电商审核场景示例**：
```
知识库内容示例：
├── 法规文档
│   ├── 《中华人民共和国广告法》全文及解读
│   ├── 《电子商务法》相关条款
│   ├── 《化妆品监督管理条例》
│   └── 《食品安全法》（食品类目）
├── 平台政策
│   ├── 抖音电商《禁售商品目录》
│   ├── 《直播行为规范》
│   ├── 《商品发布规范》
│   └── 《虚假宣传认定标准》
├── 类目知识
│   ├── 美妆：成分说明规范、功效宣称限制
│   ├── 3C数码：认证要求（3C认证）
│   ├── 食品：营养成分表规范、保质期标注
│   └── 服饰：材质说明、尺码标准
└── 判例解释
    ├── "美白"功效需特证，普通化妆品不得宣称
    ├── "原价"定义：指本次促销活动前7日内最低成交价
    └── "纯天然"不等于"有机"，需区分使用
```

**技术形态**：
- 文档切块 + Embedding + 向量数据库
- RAG检索链路
- 语义匹配（非精确匹配）

**使用方式**：模型在判断时检索相关知识作为上下文，辅助决策（"根据《广告法》第X条，该描述涉嫌..."）。

### 3. 案例库（Case Base）

**定义**：历史审核案例的**正例（合规）和反例（违规）**，用于模型微调、 few-shot提示、BadCase学习。

**电商审核场景示例**：
```
案例库条目示例：
├── 违规案例（BadCase）
│   ├── 案例ID: 2024-001
│   │   ├── 内容: "这款减肥药吃了3天瘦10斤"
│   │   ├── 违规类型: 虚假宣传+功效夸大
│   │   ├── 判定依据: 食品不得宣称减肥功效
│   │   └── 处置: 商品下架+商家警告
│   ├── 案例ID: 2024-002
│   │   ├── 内容: "全网最低价，买贵退差"
│   │   ├── 违规类型: 价格欺诈
│   │   ├── 判定依据: 无法证明"全网最低"
│   │   └── 处置: 要求修改描述
│   └── ...（更多案例）
├── 合规案例（GoodCase）
│   ├── 案例ID: 2024-100
│   │   ├── 内容: "这款面霜含烟酰胺，帮助提亮肤色"
│   │   ├── 合规点: 功效描述有科学依据，未夸大
│   │   └── 处置: 通过
│   └── ...（更多案例）
└── 模糊案例（EdgeCase）
    ├── 案例ID: 2024-200
    │   ├── 内容: "用了感觉皮肤变好了"
    │   ├── 争议点: "变好了"算功效宣称吗？
    │   └── 处置: 人工标注讨论
    └── ...（更多案例）
```

**技术形态**：
- 结构化数据库（Case ID + 内容 + 标签 + 处置）
- Few-shot Prompt的示例来源
- 模型微调的训练数据
- BadCase分析的基础

**使用方式**：
- 给模型的Few-shot示例（"以下是几个违规案例，请参考判断..."）
- 人工审核培训材料
- 模型迭代的数据基础（收集BadCase→微调模型）

### 三种库的对比

| 维度 | 规则库 | 知识库 | 案例库 |
|------|--------|--------|--------|
| **内容性质** | 结构化规则 | 非结构化文档 | 历史案例 |
| **检索方式** | 精确匹配（正则/关键词） | 语义检索（Embedding） | 标签/相似度匹配 |
| **更新频率** | 低频（法规变化） | 中频（政策调整） | 高频（每日新增） |
| **作用** | 硬规则兜底 | 提供判断依据 | 提供学习样本 |
| **容错性** | 零容忍（必须精准） | 可容忍一定噪声 | 需人工标注校准 |
| **电商场景示例** | 极限词黑名单 | 《广告法》解读文档 | "立即美白"违规案例 |

## 四、「提升电商模型内容治理能力」详解

### 什么是「内容治理能力」

在电商场景，内容治理 = **让违规内容被识别、被拦截、被处置，同时尽量减少对正常内容的误伤**。

拆解为 4 个核心指标：

```
内容治理能力
├── 识别能力（能不能找到违规）
│   ├── 准确率：判断正确的比例
│   ├── 召回率：违规内容被拦截的比例
│   └── F1值：准确与召回的平衡
├── 效率能力（审得快不快）
│   ├── 单条审核耗时（P99延迟）
│   ├── 吞吐量（每秒处理多少条）
│   └── 人机协同比例（机器审多少，人审多少）
├── 风险能力（漏放风险高不高）
│   ├── 高危违规漏放率（涉黄涉政绝不能漏）
│   ├── 舆情事件数（重大违规事件）
│   └── 人工复核覆盖率（机器低置信必过人工）
└── 体验能力（误伤多不多）
    ├── 误伤率（正常内容被误判比例）
    ├── 申诉通过率（商家申诉成功比例）
    └── 商家满意度（调研评分）
```

### 如何「提升」治理能力

**1. 准确率提升路径**
```
问题：模型把"这款面膜很补水"误判为违规（实际合规）
    ↓
分析：Case分析发现模型对"补水"和"美白"混淆
    ↓
方案：
  - 短期：Prompt里明确区分功效等级（补水是保湿，美白是功效）
  - 中期：知识库里增加功效分类说明文档
  - 长期：案例库收集更多"补水"正例，微调模型
    ↓
效果：准确率从82%提升到89%
```

**2. 召回率提升路径**
```
问题：模型漏过了"用了绝对不反弹"的极限词
    ↓
分析：规则库没有收录"不反弹"，模型也未识别
    ↓
方案：
  - 规则库：新增"绝对""不反弹"等关键词
  - 知识库：补充极限词变体表（同义/谐音/拼音）
  - 模型：收集该类BadCase，优化Prompt示例
    ↓
效果：召回率从75%提升到83%
```

**3. 效率提升路径**
```
问题：人工审核积压，直播口播延迟5分钟才出结果
    ↓
分析：当前全量人工审核，机器只过明显合规的
    ↓
方案：
  - 模型升级：提升高置信度比例（置信度>90%直接过）
  - 分层策略：高危必人工，中危抽人工，低危全自动
  - 异步改同步：流式处理替代批量处理
    ↓
效果：平均审核延迟从5分钟降到3秒，人工处理量下降60%
```

## 五、「给出解决方案」详解

### 什么是「解决方案」

不是简单说"用AI审核"，而是针对**具体业务问题**的完整可执行方案，包含：

```
解决方案标准结构
├── 问题定义
│   └── 当前痛点是什么？（人工审核慢？准确率低？成本高？）
├── 目标设定
│   └── 量化目标（审核延迟从X降到Y，准确率从A%提升到B%）
├── 方案设计
│   ├── 技术方案：Workflow设计、Prompt优化、知识库构建
│   ├── 数据方案：案例收集、标注策略、评测集构建
│   └── 运营方案：人工兜底策略、申诉流程、商家教育
├── 实施计划
│   └── 里程碑、资源需求、风险预案
├── 效果评估
│   └── 指标体系、A/B测试方案、监控告警
└── 长期迭代
    └── 持续优化机制、BadCase闭环
```

### 电商场景解决方案示例

**问题**：美妆类目虚假宣传频发，人工审核成本高，漏放导致客诉。

**解决方案**（你的输出）：

```
《美妆类目虚假宣传智能审核解决方案》

一、问题定义
- 现状：日均美妆商品上架10万件，人工审核覆盖率仅30%，虚假宣传客诉率2.3%
- 痛点：人工审不过来，模型对功效宣称识别不准

二、目标设定
- 短期（1个月）：人工审核覆盖率提升到60%，虚假宣传客诉率降到1.5%
- 中期（3个月）：覆盖率90%，客诉率<0.5%，单条审核延迟<2秒

三、方案设计

1. 知识库构建
   - 收集《化妆品功效宣称规范》《广告法》相关条款
   - 整理功效词库（美白/祛斑/防晒需特证，补水/保湿无需特证）
   - 构建判例库（1000+历史违规案例标注）

2. Workflow编排
   [商品图文输入]
       ↓
   [OCR提取文字]
       ↓
   [功效词匹配] → 命中功效词进下一步，否则快速通过
       ↓
   [知识库检索] → 召回该功效词的法规要求
       ↓
   [大模型判断] → "该商品宣称美白，是否提供特证？"
       ↓
   [置信度分层]
       ├── >90%置信度：自动通过/拦截
       ├── 50-90%置信度：人工复核队列
       └── <50%置信度：要求商家补充材料
       ↓
   [结果输出+商家通知]

3. Prompt优化
   - 结构化Prompt：角色定义+背景知识+判断标准+输出格式
   - Few-shot示例：插入5个典型违规案例和5个合规案例
   - 约束条件：不确定时标记"需人工复核"，禁止瞎猜

4. 兜底策略
   - 高危词（如"医疗效果"）100%人工复核
   - 新商家前10件商品必人工审
   - 申诉通道：商家可申诉误判，反哺模型优化

四、实施计划
- Week 1-2：知识库搭建，案例标注
- Week 3-4：Workflow开发，Prompt调试
- Week 5-6：小流量A/B测试（10%流量）
- Week 7-8：效果评估，优化迭代
- Week 9-12：逐步全量

五、效果评估
- 技术指标：准确率、召回率、F1、平均延迟
- 业务指标：人工审核覆盖率、客诉率、商家申诉率
- 成本指标：单条审核成本（人工成本+机器成本）

六、长期迭代
- 每周BadCase Review会议
- 每月模型效果复盘
- 季度知识库更新（法规变化同步）
```
