### 1.1 风险 / 补位详解（名词 + 业务理解 + 话术）

---

#### §1.1-① 顶级搜索规划 ↔ 你的 iOS 系统级入口（怎么讲才不「偏题」）

**可能风险**：面试官认为你只做「增长入口 / 客户端合作」，不熟悉 **网页搜索 / AI 搜索** 的主链路。

**业务理解（30 秒版）**  
搜索产品的核心是：**把用户的一句话需求（Query）变成可信任的结果列表或答案**，并持续用数据优化。系统级入口（Spotlight、小组件、App Intents）本质也是 **在极短路径上完成「意图识别 → 展示候选 → 用户点击/完成」**，和搜索共享同一套产品语言：**意图、转化、漏斗、延迟、bad case**。

**建议记一记的名词**


| 名词              | 通俗理解                             | 面试里怎么用                                                |
| --------------- | -------------------------------- | ----------------------------------------------------- |
| **Query**       | 用户输入的那句话/关键词                     | 「我在 iOS 项目里把用户从系统入口进来的表达，当成一种 **短 Query** 来设计承接页和埋点。」 |
| **意图 / Intent** | 用户到底想干啥（查天气、找 App、打开某功能）         | 「和搜索一样要先 **意图分类**，再决定给 **直达** 还是 **列表**。」             |
| **SERP**        | Search Engine Results Page，搜索结果页 | 「系统侧展示也可以类比 **SERP 上的卡片**，关心首条是否解决、是否还要二次点击。」         |
| **Snippet**     | 摘要条/预览文案                         | 「入口场景更要打磨 **snippet 级** 的展示，否则用户不会点进来。」               |
| **CTR**         | 点击率 = 点击 / 曝光                    | 「我在项目里用埋点看 **曝光→点击** 漏斗，逻辑和搜索 CTR 一致。」                |
| **漏斗**          | 曝光 → 点击 → 使用/完成                  | 「灰度时按版本、入口拆分漏斗，定位是 **展示问题** 还是 **承接页问题**。」            |


**承接到简历的一句话（可背）**  
「我这段经历不是传统网页搜索，但解决的是同一类问题：**在用户对时长极敏感的场景里，把意图识别清楚、把承接路径做短、用埋点验证每一层转化**——如果映射到 AI 搜索，就是 **Query 理解 → 检索/生成混合 → 结果呈现 → 用反馈迭代**。」

#### §1.1-② 搜索业务机制 + 数据策略（与 RAG 对齐，避免「只会大模型」）

**可能风险**：JD 写「理解搜索业务机制」，若你只讲 Agent，缺少 **召回、排序、索引** 等词汇。

**业务理解（主链路）**  
经典搜索大致可拆：**抓取/收录 → 建索引 → Query 理解 → 召回（多路）→ 排序（粗排/精排/重排）→ 展现策略 → 反作弊/安全**。  
AI 搜索在此基础上多一步：**何时调用生成、如何用检索结果约束生成、如何展示引用与免责声明**。

**与 RAG 的映射（面试金句）**


| 搜索环节            | 你简历里的对应物                 | 一句话                                      |
| --------------- | ------------------------ | ---------------------------------------- |
| **索引 / Index**  | 向量库 + 业务库、切块写入           | 「把可检索内容结构化进索引，和搜索建索引是同一逻辑。」              |
| **召回 / Recall** | TopK 检索、多路召回             | 「先 **多召回** 再筛选，避免『库里明明有但搜不到』。」           |
| **排序 / Rank**   | 重排模型、规则过滤、人工策略           | 「检索结果进模型前要 **rank**，否则上下文噪声会放大幻觉。」       |
| **Query 理解**    | 意图分类、是否走工具/function call | 「和 Agent 里『要不要调工具』是同一决策。」                |
| **Bad case**    | 幻觉、答非所问、引用错误             | 「和搜索里的 **零结果、摘要错误、标题党** 一样要做 case 库与归因。」 |


**建议记一记的指标（说得出定义即可）**


| 指标                    | 定义/含义             | 你怎么接简历                               |
| --------------------- | ----------------- | ------------------------------------ |
| **CTR**               | 点击/曝光             | 「入口与搜索都关心。」                          |
| **QPV / 搜索量**         | Query 次数或搜索请求量    | 「体量类指标，看需求是否被满足。」                    |
| **零结果率**              | 没有结果的 Query 占比    | 「类似我们 RAG **检索为空** 时要降级或引导改写 Query。」 |
| **首条点击率 / 首位满足**      | 第一条结果被点的比例或是否解决问题 | 「AI 搜索里常和 **首答满意度** 一起看。」            |
| **Session 深度 / 人均请求** | 一次来访搜几次、点几次       | 「反映是否一次解决问题。」                        |
| **满意度 / 人工评测**        | 抽检标注、众包打分         | 「大模型场景尤其要 **人评 + 在线信号** 双轨。」         |


**坦诚 + 积极补位（可背）**  
「我没有在百度搜索体系内做过垂直业务，但我在 RAG 里完整经历过 **索引—召回—约束生成—评测迭代**；我会用两周时间把 **粗排/精排、反作弊、时效性** 等内部文档和看板跑一遍，把名词和业务约束对齐到团队口径。」

#### §1.1-③ 跨部门协调（名词不卡壳 + 冲突故事怎么讲）

**可能风险**：只会说「我推动了」，没有 **机制词汇**。

**建议记一记的名词**


| 名词                  | 含义                                                     |
| ------------------- | ------------------------------------------------------ |
| **PRD**             | 需求文档：背景、目标、范围、非目标、交互、数据、风险                             |
| **MVP**             | 最小可用版本，先验证价值再扩展                                        |
| **里程碑 / Milestone** | 阶段交付点，用来对齐多团队节奏                                        |
| **联调**              | 多模块/多端联合调试                                             |
| **灰度 / 放量**         | 小部分用户先试，再扩大比例                                          |
| **回滚**              | 指标异常时恢复到上一版本                                           |
| **RACI**            | Responsible 执行、Accountable 拍板、Consulted 征询、Informed 知会 |
| **埋点 / Schema**     | 事件名、属性字段规范，保证数仓可对齐                                     |
| **复盘**              | 目标 vs 结果、成功因素、失败原因、下一版 action                          |


**冲突/优先级（通用 STAR 骨架，你把 iOS 或 RAG 真实细节填进去）**

- **S**：双团队资源紧，一方要「先上线入口」，一方要「等设计改版」。  
- **T**：在不变更对外承诺日期的前提下，保证核心路径可用。  
- **A**：拉会对齐 **唯一北极星指标**（如日均成功打开率）；把需求拆 **P0/P1**；P0 走现有组件，P1 后进版；书面确认 **RACI**；每日 15 分钟站会同步阻塞。  
- **R**：按期灰度；核心指标达标；P1 在 x.x 版补齐（填真实结果）。

**一句话立场**  
「跨团队我不靠硬催，靠 **共同指标 + 范围切片 + 书面共识**，冲突时用数据验证而不是嗓门。」

#### §1.1-④ 行业与竞品洞察（AI 搜索方向，各一句就够）

**可能风险**：竞品只答得出名字，没有 **差异维度**。

**对比时可用的维度（背维度比背稿更稳）**  
**答案形态**（引用/摘要/对话）、**时效与信源**、**是否多步 Agent**、**成本与延迟**、**合规与内容生态**。

**可背诵的「结论句」示例（按理解说，不必逐字）**


| 方向                                     | 观察                       | 对你有利的落点                                          |
| -------------------------------------- | ------------------------ | ------------------------------------------------ |
| **Perplexity 类**                       | 强调 **引用与可验证**，减少瞎编感      | 「我们产品上要决定 **哪些 Query 必须带来源**。」                   |
| **Google AI Overviews / Bing Copilot** | 搜索框内 **摘要 + 链接**，兼顾广告与生态 | 「平衡 **体验、商业、版权**。」                               |
| **国内 AI 搜索**                           | 更强调 **合规、可控、本土内容源**      | 「百度的优势在 **中文索引规模与实时内容**；AI 要做的是 **在可信语料上组织答案**。」 |
| **纯 Chat**                             | 强生成、弱时效                  | 「强事实问题要 **检索增强**，不能纯靠模型权重记忆。」                    |


**收束句（可背）**  
「我不认为存在单一最优形态，关键是 **按 Query 类型选『检索为主 / 生成为主 / 混合』**，并用 **指标 + bad case 库** 迭代。」

#### §1.1-⑤ 指标体系与数据洞察（无内部数据时的「方法论人设」）

**可能风险**：面试官问「你具体看过什么表」，你没有百度内部口径。

**业务理解**  
指标体系的目的：**同时衡量体验、规模、效率、风险**。AI 搜索常分三层：**在线行为指标**（快）、**质量评测**（准）、**成本指标**（省）。

**建议记一记的链路词**


| 环节  | 关键词                                    |
| --- | -------------------------------------- |
| 埋点  | **事件 event、属性 property、用户维度、版本、入口**    |
| 看板  | **漏斗、留存、分桶、趋势、同环比**                    |
| SQL | **聚合、去重、窗口、Join**（结合你 Kaggle/特征工程经历）   |
| 实验  | **AB、对照组、显著性、样本量、污染（novelty/primacy）** |
| 归因  | **假设树：是流量变了、还是转化变了、还是技术故障**            |


**无内部数据时的体面答法（可背）**  
「我没有贵司现成看板权限，但我在微博项目里从 **Schema 设计 → 埋点验收 → 漏斗分析 → 实验结论** 走过闭环；入职后我会先对齐你们 **北极星指标、护栏指标（延迟、安全、成本）** 的定义，再下钻。」

**与简历的硬连接**  
「iOS 项目里我专门管了 **Spotlight schema 与埋点策略**；RAG 项目用 **A/B 和用户反馈** 驱动迭代——这和搜索里用 **实验平台 + case 平台** 是同一套职业习惯。」

**高频追问与短答**

- **Q：怎么控幻觉？**  
**A：** 检索优先；切块与引用约束；拒答或降级策略；bad case 回流进评测集；关键操作走 function call 拉结构化数据而非纯生成。
- **Q：如何评估 Agent 好坏？**  
**A：** 任务完成率、步骤数、人工抽检、用户反馈标签；对内可再看 **采纳率 / 重复提问率**。
- **Q：和搜索的关系？**  
**A：** RAG 本质是 **检索增强生成**；搜索是「全库检索 + 排序」，我们是「限定语料 + 工具」；若做 AI 搜索，会把 **意图分类、是否触发检索、snippet 组织** 做成产品策略。

**追问**

- **Q：灰度策略？**  
**A（通用框架，可套 iOS 项目）：** ① **前置**：定义 P0 功能清单与 **准入标准**（崩溃率、核心接口成功率、关键漏斗下限）。② **分层放量**：按 **设备/OS 版本/地域/用户桶** 小比例 → 扩大；每档观察 **24～72 小时**（按业务节奏调整）。③ **对照**：保留 **对照桶** 或同期环比，避免把「外部事件」误判为「版本问题」。④ **回滚**：触发条件写清楚（如崩溃率超阈值、核心指标连续 N 小时跌破护栏、重大客诉），**一键回滚 + 归因文档**。⑤ **沟通**：每日同步研发/测试/运营，阻塞项升级。若面试官追问具体百分比，如实回答你在微博项目中的真实档位；记不清则答：**「以当时评审会定的护栏为准，我个人负责把监控看板和回滚预案写进上线 checklist。」**  
- **Q：指标跌了怎么查？**  
**A：** **漏斗拆解**（曝光→点击→完成/停留）；**维度切片**（版本、入口、机型、新老用户）；**埋点/schema 一致性**（是否漏报、字段变更）；**外部因素**（节假日、热点事件、商店审核延迟）；必要时 **分模块下线** 定位是展示层还是服务层问题。（名词表见 **§1.1-⑤**。）

### 3.3 AIGC 实习（智能体与工作流）

## 4. 岗位场景 / 业务题（AI 搜索向）

每题均可套：**用户任务 → 成功标准 → 策略（召回/排序/生成）→ 指标 → 实验 → 风险**。

### Q1：如果让你为「AI 搜索」定北极星指标，你会怎么选？

**30 秒版：** 分场景；信息型任务看 **满意答案率 / 任务完成时间**；探索型看 **会话深度与留存**。辅指标：点击率、引用点击率、差评反馈、延迟 P99。

**2 分钟版：** 先分 Query 类型（事实、复杂推理、本地、多媒体）。事实类强调 **准确与可溯源**，看 **首条满意率** 与 **人工/众包评测**；复杂任务看 **多轮解决率**。平台侧兼顾 **成本与延迟**。上线用 **AB**，防止单一 CTR 损害长期信任。

### Q2：生成和检索怎么配合？什么时候该「只搜不生」？

**A：** 强时效、强事实 —— **检索为主**，生成只做摘要与组织；开放创意 —— 生成为主但标注「非事实」。产品上要 **意图分类器** + **置信度** + **引用卡片**；你的 RAG 经验直接迁移。

### Q3：发现一个 bad case 批量出现，你怎么处理？

**A：** 聚类 Query → 看是 **召回空、排序错、摘要错、安全** 哪一类 → 定 **owner（策略/模型/前端）** → 小流量实验 → 监控指标与副作用 → 复盘文档化。

### Q4：你怎么看待竞品 AI 搜索？

**A（示例方向）：** 一类强调 **引用与可验证**；一类强调 **多步 Agent**；国内需注意 **合规与内容源**。结合百度： **搜索基建 + 大模型** 是优势，产品是 **在正确场景用混合架构**，避免为 AI 而 AI。

### Q5：跨团队推不动时怎么办？

**A：** 用 **共同指标** 说话；把需求拆 **MVP**；找 **决策人一对一** 对齐风险；书面 **RACI**；必要时 ** escalated** 用数据验证小实验争取资源。（正式面试请换你真实案例填充细节。）
