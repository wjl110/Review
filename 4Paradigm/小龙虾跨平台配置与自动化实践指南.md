# 小龙虾（Claw）个人跨平台配置、自动化实践于用户侧指南 —— 王健霖
> **能按客户的 Win / Mac 把环境配好**，然后**把不同“小龙虾”（ArkClaw / AutoClaw / QClaw / OpenClaw 生态）跑顺**；再把 **Skill + Memory** 写到位，让龙虾真能帮用户把活儿自动化干了（不只是陪聊）。  
> 注：下面默认我走的是 **OpenClaw 生态**：Skill（工具/流程）+ Memory（长期记忆）+（可选）浏览器自动化。各家产品主要差在「跑在哪儿、接在哪个 IM/平台、怎么部署」。

---

## 一、总览：产品怎么选（先把路走对）

| 产品 | 厂商 | 适用场景 | Windows | macOS | 特色/一句话 |
|------|------|----------|---------|-------|------------|
| **ArkClaw** | 火山引擎 × 飞书 | 飞书内 AI 助理、云文档协作 | 飞书客户端/网页 | 同上 | 云部署、长连接，部署很快 |
| **AutoClaw（澳龙）** | 智谱 AI | 本地桌面智能体、浏览器自动化 | ✅ .exe | ✅ .dmg | 一键安装、50+ 预置 Skill、AutoGLM |
| **QClaw** | 腾讯 | 微信内 AI 助手、桌面 Agent | ✅ Electron | ✅ .app | 微信网关、AGP（社区有人逆向） |

**客户类型速查**：

- **飞书重度、还不想运维**：优先 ArkClaw（飞书权限 + 事件订阅搞定就能用）。  
- **要本地跑、要浏览器自动化/企业内网/复用 Cookie**：优先 AutoClaw。  
- **要在微信里用**：优先 QClaw 官方路线（逆向那套只适合学习/实验，生产慎用）。  
- **多客户交付**：统一“每客户一套配置 + 目录隔离 + 环境变量”，后面省一堆扯皮。

## 二、实践案例：OpenClaw 配置的个人主页（与龙虾的协作实录）

这块已经有现成截图，讲起来很有说服力。对话截图参考：![OpenClaw](assets/OpenClaw.jpeg)

### 2.1 项目与成果

- **项目类型**：个人主页 / 作品集（Portfolio）。  
- **在线地址**：`https://wjl110.github.io/portfolio/`（GitHub Pages，Push 后一般 1–2 分钟生效）。  
- **本地路径（示例）**：
  - Windows：`C:\\Users\\<用户名>\\Documents\\portfolio`
  - macOS：`~/Documents/portfolio` 或 `~/Projects/portfolio`
- **工作流**：OpenClaw 打开项目 → 跟龙虾说“我要什么动效/布局” → 龙虾改代码并 Push → 你本地看效果继续迭代。

### 2.2 截图说明：龙虾做了啥

### 2.3 常见问题

- **GitHub Pages 没更新**：先强刷 `Ctrl+Shift+R` / `Cmd+Shift+R`；还不行就去仓库 Actions 看 Pages 构建。  
- **`API rate limit reached`**：稍后再试；建议在 SOUL 里约定“限流时怎么解释 + 给替代方案”。

## 三、实践案例：OpenClaw 去 Moltbook 发帖、留言、点赞

**Moltbook 就是 AI Agent 的社区（AI 版 Reddit）**，验证过的 Agent 才能发帖/评论/投票，人类主要围观。

### 3.1 落地链路
1. `npx molthub register` 注册我的 Agent 身份。  
2. 安装/启用 Moltbook Skill（让龙虾具备发帖/评论/点赞的能力）。  
3. 按平台要求做验证（例如 X/Twitter）。  
4. 在对话里直接下命令：  
   - “去 XX 版块发一帖：标题…内容…”  
   - “给这条帖子点个赞”  
   - “在这条帖子下面留个评论：…”  

### 3.2 限流与合规
以官方为准，常见是：发帖约每 30 分钟 1 帖，评论约每小时 50 条左右。建议把“限流自检”写进 SOUL：超限就提示稍后再试，别硬刷。

### 3.3 进阶
需要脚本化时可以用 Moltgrowth（`pip install moltgrowth`），跑 `post / upvote / comment / engage` 这类命令；日常仍用对话方式，体验更顺。

## 四、按操作系统分别配置（坑基本都在这）

### 4.1 Windows 客户环境

**交付 checklist**：

1. **安装与路径**
   - 路径尽量别中文/空格（比如 `C:\\Tools\\AutoClaw`、`C:\\Tools\\OpenClaw`）。  
   - Skill/Memory 统一用绝对路径，客户不用自己猜。

2. **权限**
   - 做浏览器自动化时别用管理员运行，不然浏览器 Profile/登录态容易对不上。

3. **网络**
   - 企业内网经常拦 WebSocket/本地服务，提前让防火墙/终端安全放行 Claw 相关进程/端口。

4. **多客户**
   - 每客户一个 `openclaw.json` 副本 + 独立 Skill/Memory 目录；密钥用环境变量，不写死文件。

**Windows 下 OpenClaw 配置示例（`openclaw.json` 片段）**：

```json
{
  "skills": {
    "load": {
      "extraDirs": ["C:\\\\Users\\\\%USERNAME%\\\\.openclaw\\\\custom_skills"],
      "watch": true
    }
  },
  "memory": {
    "path": "C:\\\\Users\\\\%USERNAME%\\\\.openclaw\\\\memory"
  }
}
```

---

### 4.2 macOS 客户环境

**Mac 上主要是权限**：

1. **安装与首次启动**
   - `.dmg` 拖到「应用程序」，若提示无法验证开发者，在「系统设置 → 隐私与安全性」里放行。

2. **权限**
   - 浏览器自动化通常需要「辅助功能」（有时还要完全磁盘访问/网络权限），没开会出现“能打开但点不了/填不了表”的情况。

3. **目录**
   - 建议统一放 `~/.openclaw/`（配置、Skill、Memory、日志），便于备份/迁移。

4. **多客户**
   - 最简单：不同 macOS 用户账号隔离。  
   - 同一账号多套配置：用 `OPENCLAW_CONFIG` 指向不同 `openclaw.json`。

**macOS 下配置示例（片段）**：

```json
{
  "skills": {
    "load": {
      "extraDirs": ["~/.openclaw/custom_skills"],
      "watch": true
    }
  },
  "memory": {
    "path": "~/.openclaw/memory"
  }
}
```

---

### 4.3 飞书 ArkClaw（与 OS 弱绑定）

ArkClaw 基本是“云上跑 + 飞书里聊”，客户侧只需要飞书客户端/网页。需要做的是：飞书应用权限、事件订阅、以及 ArkClaw 端的渠道绑定。

## 五、各款「小龙虾」的配置要点

### 5.1 飞书 ArkClaw

按这个顺序走就行：

1. 火山引擎：云服务器 → 应用模板 → 选 ArkClaw 镜像（先 1 核 2G 起）。  
2. 飞书开放平台：企业自建应用 → 机器人能力 → **批量导入权限 JSON**（官方一般会给一份）。  
3. ArkClaw 控制台：消息渠道里填飞书 **App ID / App Secret**。  
4. 事件订阅：订 `im.message.receive_v1` 等事件，走 **长连接（WebSocket）**。  
5. 发布应用：在飞书里拉机器人进群/私聊验证。

---

### 5.2 智谱 AutoClaw（澳龙）

- 安装：Windows `.exe` / macOS `.dmg`，一键装好就能用。  
- AutoGLM：适合“登录后操作、填表、抓数据”；本地跑能复用 Cookie，也能访问企业内网/VPN。  
- 自定义 Skill：把 Skill 目录加到 `extraDirs` 就能被加载。

---

### 5.3 腾讯 QClaw（含逆向注意）

- 官方方案：装客户端、绑微信、按官方能力走（最稳）。  
- 社区逆向：有人拆到微信网关协议（OAuth2 + AGP WebSocket）并开源；但合规/稳定性风险很高。  
- 结论：**学习可以，生产慎用**。

## 六、Skill 与 Memory（让龙虾“真能干活”的关键）

### 6.1 目录与配置

- Skill 从哪加载：`skills.load.extraDirs`  
- Memory 放哪：`memory.path`  
- 密钥：能用环境变量就用环境变量（别写死在配置文件里）。

### 6.2 Skill怎么写才好用

- **单一职责**：一个 Skill 就干一类事（日报/审批/填表）。  
- **强配置化**：客户 A/B 差异走 env/环境变量，不复制粘贴。  
- **错误要讲人话**：缺参/没权限/网络挂了要能定位。  
- **能脚本化就脚本化**：反复逻辑写脚本更稳。

### 6.3 Memory怎么写才不乱

- `SOUL.md`：人设、边界、做事风格（限流时怎么回复也写这）。  
- `USER.md`：客户信息/偏好/禁忌。  
- `MEMORY.md`：关键事实索引（建议 <40 行，越短越好）。  
- `memory/projects.md`：项目状态。  
- `memory/YYYY-MM-DD.md`：按日日志（记结论不记流水）。  

配 `AGENTS.md`：规定启动顺序——先 SOUL/USER，再读近期日志，需要时再读 MEMORY。

## 七、小结

先选对虾（飞书 ArkClaw / 本地 AutoClaw / 微信 QClaw 官方），再按 OS 把路径和权限这些坑堵住；Skill 写得像工具、Memory 分层不乱，最后用“个人主页 + Moltbook 社区互动”两个案例压轴，就能把实践的这套“可配置、可复用、可交付”的自动化能力讲清楚。  

- **Windows**：安装路径避免中文/空格；以当前用户运行（勿用管理员shell）；Skill/Memory 用绝对路径；防火墙放行 Claw 相关端口。
- **macOS**：`~/.openclaw/` 放配置与 Skill；辅助功能/磁盘与网络权限按需授予；多套配置可用 `OPENCLAW_CONFIG` 或不同用户区分。
- **ArkClaw**：云+飞书，客户只需飞书客户端；在火山引擎控制台完成与飞书应用的绑定。

## 实践案例

**个人主页（OpenClaw + GitHub Pages）**  
示例：`https://wjl110.github.io/portfolio/`，本地路径 Win `C:\Users\<用户>\Documents\portfolio` / Mac `~/Documents/portfolio`。流程：龙虾建仓 → 向龙虾提需求（动效/布局）→ `git push` → GitHub Pages 自动部署，1–2 分钟生效；未更新时强刷 `Ctrl+Shift+R` / `Cmd+Shift+R`。龙虾可做：粒子动画、3D 卡片悬浮、滚动渐入、渐变背景、磁力按钮、数字计数与打字机等 15+ 项动效。人设示例：SOUL 中写「天选龙虾 🦞 打工牛马 7x24」，限流时提示用户强刷或稍等。截图参考：`/Users/wjl/Downloads/OpenClaw.jpeg`。

**Moltbook 发帖、留言、点赞**  
Moltbook 为 AI Agent 社交网（AI 版 Reddit），仅验证过的 Agent 可发帖/评论/投票。流程：`npx molthub register` → 安装 Moltbook Skill → X 等验证 → 在对话中让龙虾执行「在 XX 版块发帖」「给某帖点赞」「在某帖下评论」。限流：发帖约 30 分钟 1 帖、评论约 50 条/小时；在 SOUL 中约定超限时提示用户。进阶：用 Moltgrowth（`pip install moltgrowth`）做脚本发帖/点赞/评论（`moltgrowth post`、`upvote`、`comment`、`engage`），与对话方式互补。

## Skill 与 Memory

- **配置**：`openclaw.json` 中 `skills.load.extraDirs`、`memory.path`；敏感信息用环境变量。
- **Skill**：单一职责、可配置（env/环境变量）、有错误处理与文档；常用：能力进化、摘要、Find Skills、浏览器自动化、GitHub/飞书等场景 Skill。
- **Memory**：SOUL.md（人设）、USER.md（用户）、MEMORY.md（<40 行索引）、memory/YYYY-MM-DD.md（日志）；记结论不记流水；AGENTS.md 约定先读 SOUL+USER 再日志再 MEMORY。capture/recall/consolidate 做写入、检索与定期整理。

---

## 小结

按客户 OS 与产品选型（ArkClaw / AutoClaw / QClaw）分配置与目录；用统一 Skill + Memory 规范与独立 `openclaw.json` 区分客户；个人主页走 GitHub Pages + 龙虾改站；Moltbook 走注册 + Skill + 对话/脚本，并遵守限流。即可在 Win、Mac 与多款小龙虾上实现可交付、可维护的自动化。
