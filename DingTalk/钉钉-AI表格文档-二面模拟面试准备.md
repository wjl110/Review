## 二、可能涉及的专业面问答（按类别）

### A. 项目深挖（RAG 智能体 / 微博内部提效）

**Q1：你们的 RAG 智能体整体架构是怎样的？你负责哪一块？**

- **思路**：先总后分，再强调自己负责的模块。
- **参考回答**：整体是「多模态输入 → 解析与感知 → 规划 → 决策 → 执行」的 Agent 流水线。输入支持文本、可能还有文档等；解析后进入 **Langchain** 编排的流程，规划阶段会拆任务，决策阶段会选工具（如检索、查博主库），执行阶段通过 **Function Call** 调微博内部接口。我主要负责：**RAG 链路**（切块、Embedding、向量库、检索策略与幻觉控制）、**Agent 与 LLM 的接口设计**（CoT、ReAct、多 Agent 协作）、以及**长短期记忆**的设计，让多轮对话和任务上下文能稳定复用。

**Q2：检索时如何控制幻觉？向量库选型与检索策略？**

- **参考回答**：控制幻觉主要几方面：（1）**检索阶段**：控制 top-k、阈值过滤，只把高相关片段喂给 LLM，减少「瞎编」；（2）**Prompt**：明确要求「仅根据检索内容回答，不知道就说不知道」；（3）**结果校验**：关键信息可对检索结果做一致性检查或二次检索。向量库我们用的是**内部/开源向量库**（若实际是 Milvus/ES 等可具体说），按业务做了**切块策略**（按段落/语义）、Embedding 模型选型与索引优化，检索时还会结合**关键词或元数据**做混合检索，提高召准率。

**Q3：Agent 的 Function Call 是怎么和微博内部系统对接的？有遇到性能或稳定性问题吗？**

- **参考回答**：Function Call 是 LLM 输出结构化调用意图，我们**服务端**解析后转成对微博内部**博主库、知识库等 API** 的请求。对接上做了**鉴权、限流、超时与重试**，避免内部接口被拖垮。遇到的问题包括：**高并发时 LLM 或下游接口延迟**导致超时，我们通过**异步化、队列和降级策略**（如只做检索不跑复杂推理）缓解；还有**权限与数据隔离**，保证不同部门只能访问被授权的数据。

**Q4：1500+ 人次的量级，有没有做压测或容量规划？**

- **参考回答**：有做**内部压测和灰度**。先按日活与峰值估算 QPS，对**检索、Embedding、LLM 调用**分别压测，找到瓶颈（多是 LLM 或向量检索）。再通过**缓存热点问题、限流、异步**做容量规划，并收集**真实使用反馈与监控**做迭代。若面试官追问具体数字，可按「日活约 XX，峰值 QPS 约 XX」给一个合理范围，并说明后续如何扩容（加实例、缓存、模型侧优化等）。

### B. 大模型工程与架构（贴合「AI 表格&文档」）

**Q5：如何把大模型能力接到现有文档/表格服务端里？要考虑哪些点？**

- **参考回答**：会从几层考虑：（1）**接入层**：统一 **LLM 网关**（路由、鉴权、限流、降级），支持多模型/多租户；（2）**上下文与 Token**：文档/表格内容可能很长，需要**切块、摘要、选择性注入**，控制 Token 与延迟；（3）**流式输出**：表格公式、文档续写等用 **SSE/流式** 提升体验，服务端要支持流式转发与背压；（4）**一致性与事务**：AI 改写/生成若涉及落库，要和**文档版本、协作锁**结合，避免冲突；（5）**可观测**：日志、耗时、Token 消耗、错误率，便于排障与成本控制。

**Q6：AI 表格的「智能分析」在服务端可以怎么设计？**

- **参考回答**：可以抽象成「**用户意图识别 → 数据/语义检索 → 分析/推荐 → 结果返回**」。服务端：**意图识别**可用小模型或规则+模型，低延迟；**数据侧**把表格元数据、统计信息、关键指标建索引或缓存，供检索与推荐；**分析**可以是规则引擎（如常见指标、对比）或调用 LLM 做自然语言分析；结果可以是**结构化（图表、指标）或文本说明**。同时要考虑**权限**（仅能分析用户有权限的表格）、**审计**与**限流**，避免滥用。

**Q7：流式输出（如 AI 写文档）在服务端怎么实现？要注意什么？**

- **参考回答**：一般用 **HTTP SSE 或 WebSocket**，服务端从 LLM 拉流式 chunk，边收边推给前端。注意：（1）**超时与断线**：长连接保活、断线重连与续传策略；（2）**背压**：下游消费慢时，不能无界堆积，需要流控或丢弃策略；（3）**安全与审计**：流式内容也要做敏感词、合规与日志；（4）**多端协同**：若文档是协作的，AI 写入要与 OT/CRDT 等协作模型兼容，避免冲突。

### C. 基础与架构（Java / 分布式 / 高并发 / 数据库）

*说明：你简历偏产品与 AI，若面试官考察 Java/分布式，可按「有基础 + 学习与迁移能力」来答，并尽量往你做过的「接口、协议、并发、数据」上靠。*

**Q8：Java 里 HashMap 的底层结构？为什么 8 会转红黑树？**

- **参考回答**：JDK 1.8 后是**数组 + 链表 + 红黑树**。hash 定位桶，冲突时拉链；当**单桶节点数 ≥ 8 且 数组长度 ≥ 64** 时链表转红黑树，避免极端 hash 导致链表过长，查询从 O(n) 降到 O(log n)；节点数 ≤ 6 时会退化为链表。树化阈值 8 来自泊松分布，在良好 hash 下链表长度 ≥ 8 的概率极低，用树主要是防恶意或差 hash。

**Q9：高并发下如何保证接口的幂等？**

- **参考回答**：常见做法：（1）**唯一业务键**：如订单号、请求 id，先查再插，或 DB 唯一约束；（2）**Token 机制**：先发 token，请求带 token，校验后失效，防重复提交；（3）**状态机**：只有特定状态才允许写，避免重复更新；（4）**分布式锁**：对同一业务键加锁再执行。文档/表格场景里，类似「保存」「生成」的操作都可以用「请求 id + 状态」做幂等。

**Q10：分布式场景下如何保证文档/表格的最终一致性？**

- **参考回答**：若多副本/多机房：（1）**写**：主从或多主，用**版本号/向量钟**检测冲突，冲突时提示合并或按策略解决；（2）**读**：读主或读从，从库延迟可接受时读从减轻主库压力；（3）**消息队列**：异步同步用 MQ，保证 at-least-once + 幂等消费；（4）**协作编辑**：OT 或 CRDT 保证最终一致。可以结合「钉钉文档是多人在线编辑」说：服务端要维护版本、冲突检测与合并策略，和 AI 生成内容的写入也要纳入同一套一致性模型。

**Q11：如何设计一个支持千万级用户的文档列表/表格列表接口？**

- **参考回答**：（1）**分页**：游标或 offset 分页，避免深分页；（2）**缓存**：列表摘要、权限位等用 Redis 等缓存，减轻 DB；（3）**读写分离**：读走从库或只读副本；（4）**索引**：按用户、空间、时间等建联合索引；（5）**降级**：高峰期可只返回必要字段或限制分页深度；（6）**权限**：列表过滤在服务端做，避免越权。若问「AI 相关」，可以说：AI 能力（如智能摘要、推荐）可以异步计算、结果缓存，不阻塞主列表接口。

### D. 场景与设计题

**Q12：如果让你设计「AI 表格」里「根据自然语言生成公式」的服务端模块，你会怎么设计？**

- **参考回答**：拆成几步：（1）**输入**：用户自然语言 + 当前表头/列类型/样例数据（上下文）；（2）**服务端**：接收请求后做**限流、鉴权**，组装 prompt（表结构 + 用户描述），调用 **LLM** 生成公式（如 Excel 表达式）；（3）**校验与安全**：对生成结果做**语法校验、沙箱执行**，避免恶意公式或越权引用；（4）**返回**：可流式返回公式 + 简短说明，前端再渲染预览；（5）**缓存**：相同或相似问题可缓存结果，省 Token、降延迟。存储上可记录「用户-表-问题-公式」用于迭代与审计。

**Q13：钉钉文档多人实时协作，服务端要解决哪些核心问题？**

- **参考回答**：核心包括：（1）**一致性**：用 OT 或 CRDT 把多端操作合并成一致文档；（2）**实时性**：长连接推送（WebSocket 等），操作序列化、广播、确认；（3）**冲突与合并**：同一区域并发编辑时的冲突检测与解决策略；（4）**持久化**：操作日志落库，支持历史版本与恢复；（5）**权限与可见性**：按文档权限过滤操作与推送。AI 能力（如智能续写、润色）插入的内容也要作为「操作」进入同一套协作模型，保证版本与冲突正确。

### E. 简历其他亮点（专利 / 实习 / 研究）

**Q14：MCP-A2A 协议里，和「服务端」最相关的是哪部分？**

- **参考回答**：**MCP 部分**更偏服务端：用 **JSON-RPC 2.0 over HTTP** 做标准化接口，把传感器、执行器、知识库、API 抽象成统一工具，服务端负责协议解析、路由、鉴权与上下文注入。**A2A 部分**涉及多智能体通信、任务生命周期、异步消息，服务端要维护会话、队列与状态。整体和「微服务/API 网关/消息中间件」的思路类似，都是标准化、可扩展的通信与编排。

**Q15：你觉得自己在 Java 服务端开发上是什么水平？如何补足？**

- **参考回答**：我过去更多在做 **AI 产品与模型工程**（Langchain、RAG、Agent、协议设计），对 **Java 服务端**的深度不如纯后端同学。基础方面学过 **Java、数据结构、网络与数据库**，能读能改 Java 代码，对 **MVC、接口设计、HTTP、并发** 有概念。若入职，我会**优先补**：Spring/Spring Boot 项目实践、MySQL 索引与事务、Redis 缓存与分布式锁、消息队列（Kafka/RocketMQ）在业务中的用法，并愿意从文档/表格相关需求入手，在业务里把服务端能力补齐。我的优势是把 **AI 能力合理落地到服务端架构**，和团队形成互补。
